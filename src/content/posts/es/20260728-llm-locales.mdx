Cada consulta que envías a un LLM en la nube viaja a través de la infraestructura de otra persona. El proveedor ve tus prompts, registra tus patrones y decide lo que el modelo puede y no puede decir. Obtienes conveniencia y capacidad, pero pierdes el control sobre adónde van tus datos y cómo se usan.

Ejecutar un modelo de lenguaje grande localmente invierte ese intercambio. Tus datos permanecen en tu máquina. No hay clave de API que rotar, ni endpoint de facturación del que depender, ni términos de servicio que cambien sin aviso. El modelo corre donde tú lo ejecutes.

Este post cubre por qué importa la inferencia local, qué hardware realmente necesitas y cómo poner tu primer modelo en marcha en minutos.

## Soberanía sobre la inferencia

El argumento a favor de los LLM locales no es solo privacidad. Es independencia operativa.

Una API en la nube te ofrece inteligencia bajo demanda, pero crea una cadena de dependencia: los tiempos de actividad del proveedor, sus decisiones de precios, sus políticas de retención de datos y sus filtros de contenido. Cuando OpenAI descontinuó GPT-3.5, cada aplicación que dependía de él tuvo que migrar. Cuando un proveedor cambia sus términos, te adaptas o te vas.

Un modelo local no tiene ninguna de esas restricciones. Tú eliges los pesos, tú configuras los parámetros y tú decides qué pasa con los datos. El modelo no llama a casa. No limita tu velocidad según tu nivel de suscripción. Corre hasta que tu hardware se detenga.

Esto importa para uso personal, pero importa aún más para cualquier cosa que involucre datos sensibles: documentos legales, historiales médicos, bases de código interno, análisis financieros. En el momento en que esos datos salen de tu máquina, has tomado una decisión sobre quién puede acceder a ellos.

## Qué hardware realmente necesitas

El error más común sobre LLM locales es pensar que necesitas hardware GPU caro. La realidad depende del tamaño del modelo y la herramienta que uses para ejecutarlo.

| Tamaño del Modelo | RAM Requerida | ¿GPU Útil? | Modelos de Ejemplo |
|------------------|--------------|------------|-------------------|
| 1-3B | 4-6 GB | No | Phi-3 Mini, TinyLlama |
| 7-8B | 8-12 GB | Opcional | Llama 3 8B, Mistral 7B, Qwen2 7B |
| 13-14B | 16-24 GB | Sí | Llama 3 13B, Mistral 14B |
| 30-34B | 32-48 GB | Sí | CodeLlama 34B, Qwen2 32B |
| 70B+ | 64+ GB | Sí (multi-GPU) | Llama 3 70B, Mixtral 8x7B |

La variable clave es la RAM. Un modelo de 7B cuantizado a 4 bits necesita aproximadamente 5-6 GB de RAM. Si tienes 16 GB de RAM en tu portátil, puedes ejecutar un modelo de 7B cómodamente mientras mantienes el sistema responsive.

La aceleración GPU importa para la velocidad, no para la capacidad. Un setup solo con CPU genera aproximadamente 5-10 tokens por segundo en un modelo de 7B. Una GPU de consumo como una RTX 3060 con 12 GB de VRAM eleva eso a 30-50 tokens por segundo. Ambos son utilizables. La GPU hace que la experiencia se sienta instantánea en vez de tolerable.

Los Macs con Apple Silicon son un caso especial. La arquitectura de memoria unificada significa que CPU y GPU comparten el mismo pool de RAM. Un MacBook Pro con 32 GB de memoria unificada puede ejecutar un modelo de 13B a velocidades de GPU sin gráficos discretos.

## Ollama: el inicio en cinco minutos

Ollama es la forma más sencilla de ejecutar un LLM local. Maneja descargas de modelos, cuantización y servicio con un solo binario.

```bash
# Instalar Ollama (macOS, Linux, Windows)
curl -fsSL https://ollama.ai/install.sh | sh

# Descargar y ejecutar un modelo
ollama run llama3.1:8b

# Iniciar un servidor API (compatible con OpenAI)
ollama serve
```

El comando `ollama run` descarga el modelo en la primera ejecución y luego te lleva a un chat interactivo. El modelo permanece cargado en memoria para consultas posteriores.

Ollama expone una API compatible con OpenAI en el puerto 11434 por defecto. Cualquier herramienta que funcione con la API de OpenAI puede apuntar a tu instancia local:

```bash
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "llama3.1:8b",
    "messages": [{"role": "user", "content": "Explica el teorema CAP en un párrafo."}]
  }'
```

La compatibilidad con OpenAI no es casualidad. Es la razón por la que Ollama se ha convertido en el backend por defecto para herramientas de IA local. Cada herramienta del ecosistema puede conectarse a él sin modificaciones.

## llama.cpp: la alternativa ligera

llama.cpp es una implementación en C++ de la arquitectura de modelos Llama sin dependencias más allá de un compilador de C++. Corre en CPU, con aceleración GPU opcional vía Metal (macOS), CUDA (NVIDIA) o Vulkan.

```bash
# Compilar desde el código fuente
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
cmake -B build
cmake --build build --config Release

# Descargar un modelo cuantizado (formato GGUF)
wget https://huggingface.co/TheBloke/Llama-2-7B-Chat-GGUF/resolve/main/llama-2-7b-chat.Q4_K_M.gguf

# Ejecutar inferencia
./build/bin/llama-cli -m llama-2-7b-chat.Q4_K_M.gguf \
  -p "¿Cuál es la diferencia entre TCP y UDP?" \
  -n 256
```

llama.cpp es el motor detrás de la mayoría de herramientas de IA local. Ollama lo usa internamente. LM Studio lo usa. GPT4All lo usa. Entender qué pasa bajo techo te da control sobre la cuantización, la longitud del contexto y la asignación de hilos.

El formato GGUF es el estándar para modelos cuantizados. La convención de nombres te dice el nivel de cuantización: Q4_K_M significa cuantización de 4 bits con calidad media. Q5_K_M ofrece mejor calidad a costa de más RAM. Q8_0 es calidad casi original pero usa aproximadamente el doble de memoria.

## Elegir tu primer modelo

El panorama de modelos se mueve rápido, pero algunas familias se han demostrado para uso local:

- **Llama 3.1 (Meta)** — La familia más versátil. La variante de 8B corre bien en hardware de consumo. La de 70B compite con modelos de clase GPT-4. Fuerte en instrucciones, razonamiento y código.
- **Mistral / Mixtral** — Excelente en seguimiento de instrucciones con un footprint menor. El modelo de 7B supera su peso. Mixtral 8x7B usa una arquitectura de mixture-of-experts que ofrece calidad de clase 14B a costo de inferencia de clase 7B.
- **Qwen 2** — Rendimiento multilingüe fuerte, particularmente bueno en chino e inglés. Las variantes de 7B y 32B están bien cuantizadas en repositorios de la comunidad.
- **Phi-3 (Microsoft)** — Diseñado para rendimiento en modelos pequeños. El modelo de 3.8B corre en teléfonos y portátiles con 4 GB de RAM. Bueno para tareas ligeras donde un modelo más grande es excesivo.
- **CodeLlama / DeepSeek Coder** — Especializado en generación de código. Si tu caso de uso principal es asistencia de programación, estos modelos superan a alternativas generales con el mismo conteo de parámetros.

Empieza con Llama 3.1 8B o Mistral 7B. Son el mejor equilibrio de calidad, velocidad y requisitos de hardware para alguien que empieza.

## Primeros pasos

La barrera para ejecutar un LLM local es más baja que nunca. Ollama te da un modelo funcionando en cinco minutos. llama.cpp te da control sin overhead. Los modelos mismos son gratuitos y mejoran en un ciclo mensual.

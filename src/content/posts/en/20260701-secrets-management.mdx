GitOps made the cluster state auditable by storing every Kubernetes manifest in Git. But manifests often reference secrets — database passwords, API tokens, TLS certificates. If those secrets end up in the repository in plain text, the audit trail is worthless because the repository itself becomes a vulnerability.

The solution is not to stop using GitOps. It is to keep secrets out of Git entirely and let the cluster fetch them from a trusted external store at sync time.

## External Secrets Operator

External Secrets Operator (ESO) is a Kubernetes controller that synchronises secrets from external APIs into native Kubernetes Secret objects. You define what to fetch and where to store it, and ESO keeps them in sync.

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: database-credentials
  namespace: api
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secretsmanager
    kind: SecretStore
  target:
    name: database-credentials
  data:
  - secretKey: password
    remoteRef:
      key: production/api/database
      property: password
```

When this ExternalSecret is applied, ESO reads the key `production/api/database` from AWS Secrets Manager, extracts the `password` property, and creates a Kubernetes Secret named `database-credentials`. The process repeats every hour, so rotating the secret in the external store propagates to the cluster automatically.

ESO supports multiple backends: AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, HashiCorp Vault, and others. The SecretStore resource abstracts the backend configuration. An ClusterSecretStore makes it available cluster-wide.

The GitOps repository contains the ExternalSecret definition. It does not contain the actual secret value. ArgoCD syncs the ExternalSecret, ESO fills in the values from the external store, and the pod mounts the resulting Secret. The secret material never touches the repository.

## SOPS with Age

ESO requires an external service — a cloud provider or Vault. For smaller teams or homelab clusters, that may be too much infrastructure. SOPS (Secrets OPerationS) encrypts individual values or entire files using asymmetric encryption. The encrypted file is safe to commit to Git.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: api-key
type: Opaque
stringData:
  #ENC[AES256_GCM,data:encrypted-base64,iv:...,tag:...]
```

SOPS encrypts only the values, leaving the structure readable. Age is the recommended encryption key format — a single small key pair stored outside the repository. The CI pipeline or ArgoCD decrypts the file before applying it.

For ArgoCD to work with SOPS, you configure a plugin or use the argocd-vault-plugin, which decrypts SOPS-encrypted files at sync time. The private key is stored as a Kubernetes Secret in the cluster, not in Git.

SOPS does not support automatic rotation. To rotate a secret, you decrypt the file, update the value, and re-encrypt it. The process is manual, which makes it suitable for static secrets — API tokens that change once a quarter — but not for dynamic secrets with short lifetimes.

## HashiCorp Vault

Vault handles dynamic secrets — credentials that are generated on demand and expire after a configurable time. A database password generated by Vault is valid for 24 hours and then disappears. If it leaks, the window of exposure is bounded.

```yaml
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: vault-backend
spec:
  provider:
    vault:
      server: https://vault.internal:8200
      path: secret
      version: v2
      auth:
        kubernetes:
          mountPath: kubernetes
          role: api-server
```

Vault authenticates via Kubernetes service accounts — a pod proves its identity and receives a token for the specific secret path it is allowed to read. There is no long-lived credential to leak.

The operational cost is higher. Vault is a stateful service that requires storage, backup, and a high-availability configuration. For production clusters with many services and frequent rotations, it is the right tool. For a single-team cluster with a handful of static secrets, it is overkill.

## Choosing the Right Tool

The three approaches solve the same problem at different scales.

| Tool | Secret location | Rotation | Setup cost | Best for |
|------|----------------|----------|------------|----------|
| ESO | External store (cloud) | Automatic on store update | Low | Teams already using a cloud provider |
| SOPS + age | Encrypted in Git | Manual | Minimal | Small teams, homelab, static secrets |
| Vault | Vault storage | Dynamic, scheduled | High | Multi-team, compliance, short-lived creds |

All three integrate with ArgoCD. The choice depends on how many secrets you manage, how often they rotate, and what infrastructure you already run.

## What Not to Do

Kubeseal is a popular alternative that encrypts secrets into SealedSecret resources. The sealed data is safe to commit, and the controller inside the cluster decrypts it. The problem is that SealedSecrets are tied to a specific cluster and namespace. If you rebuild the cluster, the encryption key changes and all sealed secrets become invalid. Disaster recovery requires backing up the sealing key, which reintroduces the bootstrapping problem.

Storing secrets in a separate Git repository with restricted access is another common pattern. It works until someone adds a CI pipeline that reads from that repository and exposes the secret in logs or build artifacts. The history is also permanent — a deleted secret remains in the Git history forever.

## The Principle

Secrets should never be stored in the same repository as the configuration that consumes them. They should be fetched from a system designed for secret storage — a cloud secrets manager, an encrypted file, or Vault. The GitOps repository commits to where the secret lives, not what it contains. The cluster resolves the reference at runtime, and the secret stays where it belongs.

El post anterior sobre GitOps terminaba con una advertencia: el repositorio de Git no debe contener contraseñas ni claves en texto plano. Pero en la práctica, casi cualquier aplicación necesita credenciales para funcionar. Una base de datos, un API de terceros, un certificado TLS. Si esas credenciales acaban en el repositorio, el modelo GitOps pierde su principal ventaja: que el estado del clúster sea seguro y auditable.

La solución no es usar GitOps a medias. Es mantener los secretos fuera de Git y dejar que el clúster los obtenga desde un almacén externo en el momento de la sincronización.

## External Secrets Operator

External Secrets Operator (ESO) es un controlador de Kubernetes que sincroniza valores desde APIs externas y los convierte en objetos Secret nativos. Tú defines qué quieres obtener y de dónde, y ESO se encarga del resto.

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: database-credentials
  namespace: api
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secretsmanager
    kind: SecretStore
  target:
    name: database-credentials
  data:
  - secretKey: password
    remoteRef:
      key: production/api/database
      property: password
```

Cuando este ExternalSecret se aplica al clúster, ESO consulta la clave `production/api/database` en AWS Secrets Manager, extrae el campo `password` y crea un Secret de Kubernetes con ese valor. El proceso se repite cada hora. Si rotas la contraseña en el almacén externo, el clúster la recoge automáticamente en la siguiente sincronización.

ESO funciona con múltiples backend: AWS Secrets Manager, GCP Secret Manager, Azure Key Vault y HashiCorp Vault, entre otros. El recurso SecretStore abstrae la configuración del backend. ClusterSecretStore lo hace disponible en todo el clúster.

El repositorio GitOps contiene la definición del ExternalSecret, pero nunca el valor real. ArgoCD sincroniza el ExternalSecret, ESO rellena los valores desde el almacén externo, y el pod monta el Secret resultante. El material sensible no toca Git en ningún momento.

## SOPS con Age

ESO requiere un servicio externo — un proveedor cloud o Vault. Para equipos pequeños o clústeres domésticos, montar ese servicio es más infraestructura de la que se quiere gestionar. SOPS (Secrets OPerationS) cifra valores individuales o archivos completos usando criptografía asimétrica. El archivo cifrado puede ir a Git sin riesgo.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: api-key
type: Opaque
stringData:
  #ENC[AES256_GCM,data:base64-cifrado,iv:...,tag:...]
```

SOPS cifra solo los valores. La estructura del archivo sigue siendo legible. Age es el formato de clave recomendado — un par de clave pequeño que se guarda fuera del repositorio. El pipeline de CI o ArgoCD descifra el archivo antes de aplicarlo.

Para que ArgoCD funcione con SOPS, se configura el argocd-vault-plugin, que descifra los archivos SOPS en el momento de la sincronización. La clave privada se almacena como un Secret de Kubernetes en el clúster fuera de Git.

SOPS no permite rotación automática. Para rotar un secreto, descifras el archivo, cambias el valor y lo vuelves a cifrar. El proceso es manual. Es útil para secretos estáticos — claves de API que cambian una vez al trimestre — pero no para credenciales dinámicas con vida corta.

## HashiCorp Vault

Vault gestiona secretos dinámicos: credenciales que se generan bajo demanda y expiran tras un tiempo configurable. Una contraseña de base de datos generada por Vault vale 24 horas y luego desaparece. Si se filtra, la ventana de exposición está acotada.

```yaml
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: vault-backend
spec:
  provider:
    vault:
      server: https://vault.internal:8200
      path: secret
      version: v2
      auth:
        kubernetes:
          mountPath: kubernetes
          role: api-server
```

Vault se autentica mediante service accounts de Kubernetes. Un pod demuestra su identidad y recibe un token para una ruta de secretos concreta. No hay una credencial de larga duración que pueda filtrarse.

El coste operativo es alto. Vault es un servicio stateful que necesita almacenamiento, backups y una configuración de alta disponibilidad. Para clústeres de producción con muchos servicios y rotaciones frecuentes, es la herramienta adecuada. Para un clúster monoequipo con cuatro secretos estáticos, sobra.

## Cómo elegir

Las tres opciones resuelven el mismo problema a escalas distintas.

| Herramienta | Dónde está el secreto | Rotación | Coste inicial | Para qué sirve |
|-------------|----------------------|----------|---------------|----------------|
| ESO | Almacén externo (cloud) | Automática al cambiar | Bajo | Equipos que ya usan un proveedor cloud |
| SOPS + age | Cifrado en Git | Manual | Mínimo | Equipos pequeños, homelab, secretos estáticos |
| Vault | Almacén de Vault | Dinámica, programada | Alto | Varios equipos, compliance, credenciales cortas |

Las tres se integran con ArgoCD. La elección depende de cuántos secretos gestiones, cada cuánto rotan y qué infraestructura tienes ya montada.

## Lo que no funciona

Kubeseal cifra secretos en recursos SealedSecret que pueden ir a Git. El controlador dentro del clúster los descifra. El problema es que SealedSecrets están atados a un clúster y un namespace concretos. Si reconstruyes el clúster, la clave de cifrado cambia y todos los SealedSecrets se vuelven inválidos. Para recuperarlos necesitas la clave de sellado, lo que reintroduce el problema de arranque.

Guardar secretos en un repositorio Git separado con acceso restringido es otro patrón habitual. Funciona hasta que alguien añade un pipeline que lee de ese repositorio y expone el secreto en los logs o en los artefactos de build. El historial de Git es permanente: un secreto borrado sigue ahí para siempre.

## El principio

Los secretos no deben estar en el mismo repositorio que la configuración que los consume. Deben obtenerse de un sistema diseñado para almacenar secretos: un gestor cloud, un archivo cifrado o Vault. El repositorio GitOps declara dónde vive el secreto, no lo que contiene. El clúster resuelve la referencia en tiempo de ejecución, y el secreto se queda donde debe estar.

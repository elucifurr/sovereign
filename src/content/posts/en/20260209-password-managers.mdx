Password managers are usually marketed as convenience tools. They remember your passwords so you do not have to. They generate strong random passwords. They auto-fill login forms. These are genuine benefits, but they describe the password manager as a productivity tool, not as what it really is — critical infrastructure for digital autonomy.

Your password manager holds the keys to every account you own. Email, banking, social media, cloud infrastructure, domain registration — all of it is behind credentials that the password manager stores. The security of your entire digital life depends on the security of that single database. This makes the choice of password manager a sovereignty decision.

## The Local-First Model

The security model of a password manager depends on where and how it stores your encrypted vault. Local-first password managers like KeePassXC store the vault as a file on your own device. The vault never touches a server unless you explicitly sync it.

This model gives you complete control. You choose where to store the vault file, how to back it up, and how to sync it between devices — if at all. The vault is encrypted with a master password and optionally a key file. No third party has access to the encrypted data.

The trade-off is convenience. Syncing between devices is manual unless you set up your own sync mechanism. There is no web interface. If you lose the vault file and the backup, you lose access to all your credentials. The responsibility is entirely yours.

## The Self-Hosted Model

Self-hosted password managers like Vaultwarden (a Bitwarden-compatible server written in Rust) provide the convenience of a cloud service without the dependency on a third party. You run the server on your own infrastructure, and the clients connect to it.

Vaultwarden is a lightweight implementation of the Bitwarden server API. It uses under 10 MB of memory and runs on any system with Rust support. The clients — desktop, mobile, browser extension — are the same Bitwarden clients used by hundreds of thousands of people. The difference is that the server is yours.

The security model is similar to the local-first model with one addition: the server holds encrypted vault data that can be synced between devices automatically. The server never sees the master password or the unencrypted vault. Encryption and decryption happen on the client.

Self-hosting a password manager requires maintaining a server and keeping it accessible. This is not a trivial requirement, but it is within the capabilities of anyone who already runs a VPS. The operational overhead is low once the server is running.

## The Cloud Model

Cloud password managers like 1Password, Bitwarden's hosted service, and Dashlane offer the maximum convenience. Vaults are synced automatically, available from any device, and backed up by the provider. The user does not need to manage infrastructure.

The trade-off is trust. The provider could be compelled to serve a malicious update that exfiltrates the vault. The provider could be hacked. The provider could change its pricing, terms of service, or feature set. The user has no recourse except to switch providers, which means exporting and importing vaults — a process that is rarely seamless.

Cloud password managers are better than no password manager. For most users, they are the right choice. But the convenience comes with a dependency relationship that the local-first and self-hosted models avoid.

## The Threat Model

Choosing a password manager requires understanding the threat model. A cloud password manager protects against the most common threats — credential reuse, phishing, weak passwords. A self-hosted or local-first password manager adds protection against threats from the provider itself.

The difference matters when the threat is not a random attacker but a compromise of the service provider. Cloud password managers have been targeted in phishing attacks, social engineering attacks, and supply chain attacks. The vault encryption protects the contents, but the metadata — which services you use, when you access them — is visible to the provider.

A self-hosted password manager behind a VPN or on a local network eliminates most of these risks. The attack surface is limited to the server you control and the clients you install. There is no third party to phish, no central database to breach, no support team to social-engineer.

## The Setup Cost

Setting up Vaultwarden takes about thirty minutes. Download the binary, create a configuration file, set up a reverse proxy with TLS, and start the service. The clients connect to your server URL and work exactly as they would with the hosted service.

The ongoing maintenance is minimal. Vaultwarden receives regular updates that fix bugs and security issues. Updating means downloading a new binary and restarting the service. Backups mean copying the vault database to another location.

Password managers are not optional infrastructure. Using the same password across services is irresponsible. Using weak passwords is irresponsible. The only question is whether you want the convenience of a cloud service or the autonomy of a self-hosted solution.

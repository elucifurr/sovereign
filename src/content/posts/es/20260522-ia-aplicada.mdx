## La pregunta de la autonomía

Cada herramienta de IA incorpora un intercambio. Ganas capacidad — código más rápido, mejor prosa, información extraída de grandes volúmenes de datos — y pierdes algo a cambio. Eso que pierdes depende de cómo está construida la herramienta y de quién la controla.

La pregunta de la autonomía no es si la IA es útil. Es si la herramienta aumenta o disminuye tu capacidad para tomar decisiones independientes sobre tus propios sistemas.

Una API de LLM alojada en la nube te da inteligencia bajo demanda, pero crea una dependencia de un proveedor específico, sus precios, sus políticas de datos y su existencia continuada. Un modelo que ejecutas localmente te da menos capacidad bruta, pero preserva tu capacidad de operar sin pedir permiso.

Este artículo trata de navegar ese intercambio de forma deliberada.

## El espectro de la dependencia

Resulta útil situar las herramientas de IA en un espectro de autonomía:

| Tipo de herramienta | Ejemplo | Nivel de autonomía |
|--------------------|---------|-------------------|
| API en la nube (propietaria) | GPT-4, Claude API | Bajo |
| API en la nube (pesos abiertos) | Llama vía Groq, Together | Medio |
| Modelo pequeño autoalojado | Llama 3B vía Ollama | Alto |
| Modelo grande autoalojado | Llama 70B vía Ollama + GPU | Máximo (pero caro) |
| Herramientas locales con IA | Copilot con backend local | Medio-Alto |
| Flujos de IA que controlas | Pipelines personalizados con inferencia local | Máximo |

El patrón es claro: la autonomía se correlaciona con cuánta infraestructura controlas.

## IA local práctica

La IA local ha mejorado drásticamente. Lo que era inviable hace seis meses ahora es una herramienta de uso diario.

### Asistencia al código

El caso de uso más impactante para la mayoría de desarrolladores es la codificación asistida por IA. La cuestión es si el asistente depende de ti o de un proveedor en la nube.

**Continue.dev** es un asistente de código abierto que se conecta a cualquier backend. Puedes combinarlo con:

```bash
# Ejecutar un modelo local vía Ollama
ollama run codellama:7b-instruct

# O usar un modelo más grande si tienes el hardware
ollama run deepseek-coder:33b
```

La experiencia no es idéntica a GitHub Copilot — la velocidad de completado y la calidad de las sugerencias varían — pero los datos nunca salen de tu máquina. Para muchas tareas, la diferencia es insignificante.

Para tareas más pesadas como refactorizar entre archivos o generar código repetitivo, un modelo local de 7B parámetros maneja la mayor parte de lo que un desarrollador necesita. La latencia es mayor que en la nube, pero el techo (sin límites de tasa, sin coste, sin registro) lo compensa.

### Escritura e investigación

Para prosa, los modelos locales están por detrás de los de nube, pero alcanzándolos rápidamente. Un Llama 3 8B o Mistral 7B local produce borradores utilizables para escritura técnica, correos electrónicos y documentación.

El flujo de trabajo que utilizo:

1. Escribo un esquema aproximado manualmente
2. Pido al modelo local que desarrolle una sección
3. Edito intensamente — el modelo es un compañero de borrador, no un autor
4. Ejecuto una revisión gramatical local (LanguageTool vía servidor autoalojado)

Cada paso me mantiene al control del resultado final. La IA acelera las partes mecánicas; la estructura, la voz y las decisiones siguen siendo mías.

**La heurística importante:** si no puedes explicar lo que dice un párrafo generado por IA, no deberías publicarlo. Entender es el precio de la propiedad.

### Trabajo con conocimiento

La generación aumentada por recuperación (RAG) local es donde la IA autoalojada brilla. Herramientas como **Ollama + AnythingLLM** o **LocalAI** te permiten indexar tus propios documentos — manuales técnicos, notas de proyectos, fragmentos de libros — y consultarlos sin enviar datos a ningún sitio.

```bash
# Configuración RAG mínima con Ollama
ollama pull nomic-embed-text
ollama pull llama3:8b

# Usa AnythingLLM o un pipeline personalizado
# para incrustar y consultar tus documentos locales
```

Esto es genuinamente mejor que las alternativas en la nube para conocimiento sensible o especializado. Tu documentación interna, wiki personal o base de código se mantiene privada. El modelo responde en tu contexto, no en el de otro.

## Cuándo tiene sentido la nube

Hay razones legítimas para usar IA en la nube. La clave es elegirlas deliberadamente en lugar de por defecto:

- **Tareas limitadas por hardware.** Entrenar o ajustar modelos grandes requiere GPUs que la mayoría de individuos no poseen. Usar computación en la nube para tareas específicas y acotadas es racional — siempre que controles el ciclo de vida de los datos.
- **Prototipado.** Al explorar un dominio nuevo, las APIs en la nube te permiten probar hipótesis antes de comprometerte con infraestructura local.
- **Tareas puntuales.** Traducir un documento, resumir un artículo o generar un script de un solo uso. Si los datos no son sensibles y la tarea es transitoria, la nube es pragmáticamente aceptable.

La regla: usa la IA en la nube como una herramienta que coges y sueltas, no como infraestructura sobre la que construyes.

## Señales de alarma

Ciertos patrones indican que la herramienta está socavando tu autonomía:

- **Sin exportación.** Si no puedes llevarte tus conversaciones, modelos ajustados o datos de uso a otro sitio, estás atrapado.
- **Dependencia de API.** Si tu flujo de trabajo se rompe por completo cuando un proveedor cae o cambia sus precios, tienes un punto único de fallo.
- **Entrenamiento con tus datos.** Si el proveedor entrena con tus entradas, cada mensaje que escribes se convierte en parte de su producto. Asume que lo hacen salvo que demuestren lo contrario.
- **Opacidad.** Si no puedes saber por qué el modelo produjo una salida determinada, no puedes auditarlo. En sistemas que afectan decisiones, la auditabilidad no es opcional.

## Construye tu pila

Una pila de IA que preserve la soberanía se ve así, de más a menos importante:

1. **Un plan de contingencia local.** Incluso si usas APIs en la nube para tareas pesadas, mantén un modelo local que pueda manejar tu flujo de trabajo principal. Ollama con un modelo de 7B parámetros funciona en cualquier máquina con 8 GB de RAM.
2. **Pesos abiertos.** Prefiere modelos con pesos publicados (Llama, Mistral, DeepSeek, Qwen) sobre APIs opacas. Los pesos significan que puedes ejecutar el modelo tú mismo y verificar su comportamiento.
3. **Contexto local.** Mantén tus datos en local. Usa pipelines RAG que indexen tus archivos sin subirlos.
4. **Interfaces estándar.** Usa herramientas que soporten APIs compatibles con OpenAI para poder cambiar de backend sin cambiar tus herramientas.

```bash
# Una prueba sencilla: ¿puedes hacer tu trabajo asistido por IA
# sin acceso a internet durante un día?
# Si no, tu pila tiene un problema de dependencia.
```

## Conclusión

La IA aplicada no está inherentemente reñida con la autonomía personal. El conflicto está entre el uso intencional y el consumo pasivo, entre el control local y la dependencia remota, entre herramientas que operas y herramientas que te operan.

El mismo principio aplica aquí que en cualquier otra capa de la pila: sabe de qué dependes y asegúrate de que la dependencia es una elección, no un estado por defecto.

Empieza con un modelo local. Entiende sus límites. Añade servicios en la nube solo cuando el intercambio sea claro y reversible. El objetivo no es rechazar la IA, sino usarla en tus propios términos.
